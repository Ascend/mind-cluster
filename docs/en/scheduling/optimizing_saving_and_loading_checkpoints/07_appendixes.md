## (Optional) Using DPC File System to Access Storage to Accelerate Checkpoint Loading

Check whether the following conditions are met:

- Whether the DPC (Distributed Parallel Client) file system is used to access storage.
- Whether the NDS 1.0 package (`/opt/oceanstor/dataturbo/sdk/lib/libdpc_nds.so`) is successfully installed.
- Whether the training process (if inside a container) can access this `.so` file.

If all the above conditions are met, the NDS 1.0  direct‑pass read function is automatically enabled to accelerate checkpoint loading.

The criterion for determining whether NDS 1.0 is successfully loaded is to check whether the following message appears in the log:

```text
"initial and open nds file driver success"
```

For details about NDS 1.0, see the [*OceanStor DataTurbo 25.x.x DTFS User Guide*](https://support.huawei.com/enterprise/en/doc/EDOC1100539414/3f076df0/restoring-a-client?idPath=7919749|251366268|250389224|263153904|264568316).

> [!CAUTION] CAUTION
> If you use the DPC to access storage and have successfully installed NDS 1.0 (installation path: `/opt/oceanstor/dataturbo/sdk/lib/libdpc_nds.so`), setting the permission to `444` ensures normal functionality. Before starting training, exercise caution when setting the permissions for this file.

## Setting User Expiration

To ensure user security, set the user expiration date via the system command `chage`.

Example:

```bash
chage [-m mindays] [-M maxdays] [-d lastday] [-I inactive] [-E expiredate] [-W warndays] user
```

For details about the parameters, see [Table 1](#table_acp_04).

**Table 1<a id="table_acp_04"></a>**  Setting user expiration

|Parameter|Description|
|--|--|
|-d<br>--lastday|Date of the last change.|
|-E<br>--expiredate|Date on which the user account expires. After this date, the user will be unavailable.|
|-h<br>--help|Displays command help information.|
|-i<br>--iso8601|Changes the expiration date of the user password and displays it in "YYYY-MM-DD" format.|
|-I<br>--inactive|Inactive period. After the specified number of days, the password is set to an invalid state.|
|-l<br>--list|Lists current settings. Used by non-privileged users to determine when their password or account expires.|
|-m<br>--mindays|Minimum number of days before a password can be changed. Setting this to "0" means the password can be changed at any time.|
|-M<br>--maxdays|Maximum number of days a password remains valid. Setting this to "-1" removes the password validity check. Setting this to "99999" means unlimited.|
|-R<br>--root|Sets the root directory for command execution to the specified directory.|
|-W<br>--warndays|Number of days in advance a warning is issued before the user's password expires.|

> [!NOTE]
>
> - The date format is "YYYY-MM-DD". For example, `chage -E 2017-12-01 test` indicates that the password for user `test` expires on December 1, 2017.
> - `user` is mandatory. Replace it with the specific user; defaulted to `root`.
> - Account passwords should be updated regularly; otherwise, it may lead to security risks.

Example: Change the validity period of user `test` to 90 days.

```bash
chage -M 90 test
```

## Password Complexity Requirements

Passwords must meet at least the following requirements:

1. The password must be at least 8 characters long.
2. The password must contain a combination of at least two of the following character types:
    - One lowercase letter
    - An uppercase letter
    - A digit
    - A special character: `~!@#$%^&*()-_=+\|[{}];:'",<.>/? and space

3. The password cannot be the same as the username.
